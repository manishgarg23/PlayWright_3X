# Source Code vs Byte Code vs Machine Code

## Quick Summary Table

| Feature | Source Code | Byte Code | Machine Code |
|---|---|---|---|
| Written by | Humans | Compiler/Transpiler | Compiler/Assembler |
| Readable | Yes (human-readable) | Partially (structured binary) | No (raw binary) |
| Language | High-level (JS, Python, Java) | Intermediate (JVM, V8 IR) | Binary (0s and 1s) |
| Portability | Portable | Portable (with runtime) | NOT portable (CPU-specific) |
| Execution speed | Slowest (needs compilation) | Medium | Fastest |
| File example | `.js`, `.py`, `.java` | `.class`, `.pyc` | `.exe`, `.bin` |

---

## 1. Source Code

### What is it?
Source code is the **human-written instructions** in a programming language. It is text that developers read, write, and maintain.

### Characteristics
- Written in high-level languages (JavaScript, Python, Java, C++)
- Fully human-readable
- Not directly executable by a CPU
- Must be compiled, interpreted, or transpiled before execution

### Example — JavaScript Source Code
```javascript
// source code: hello.js
function greet(name) {
    return "Hello, " + name + "!";
}

console.log(greet("World"));
```

### Example — Python Source Code
```python
# source code: hello.py
def greet(name):
    return f"Hello, {name}!"

print(greet("World"))
```

### Example — Java Source Code
```java
// source code: Hello.java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
```

### Key Points
- This is what you push to GitHub
- This is what code reviews are done on
- Abstraction level is HIGH — you think in logic, not memory addresses

## 2. Byte Code

### What is it?
Byte code is an **intermediate representation** of your source code. It sits between human-readable source code and raw machine code. It is generated by a compiler and executed by a **Virtual Machine (VM)** or **runtime**.

### Characteristics
- Not human-readable (structured binary format)
- Platform-independent — runs on any machine that has the correct runtime/VM
- Faster than interpreting source code directly
- Examples: JVM bytecode (`.class`), Python bytecode (`.pyc`), V8 IR (Node.js)

### How It Works

```
Source Code  -->  [Compiler]  -->  Byte Code  -->  [VM / Runtime]  -->  Execution
hello.java         javac          hello.class        JVM
hello.py           CPython        hello.pyc          Python VM
hello.js           V8 engine      (in-memory IR)     V8
```

### Example — Java Byte Code (disassembled with `javap -c`)
```
// Compiled from "Hello.java"
public class Hello {
  public static void main(java.lang.String[]);
    Code:
       0: getstatic     #2   // Field java/lang/System.out
       3: ldc           #3   // String "Hello, World!"
       5: invokevirtual #4   // Method java/io/PrintStream.println
       8: return
}
```
> This is the **byte code** — instructions for the JVM, not a CPU.

### Example — Python Byte Code (using `dis` module)
```python
import dis

def greet(name):
    return "Hello, " + name + "!"

dis.dis(greet)
```
Output:
```
  2           0 LOAD_CONST               1 ('Hello, ')
              2 LOAD_FAST                0 (name)
              4 BINARY_ADD
              6 LOAD_CONST               2 ('!')
              8 BINARY_ADD
             10 RETURN_VALUE
```
> Each line is a **byte code instruction** for Python's virtual machine.

### Key Points
- Byte code is tied to a **runtime/VM**, not a hardware CPU
- `.pyc` files in Python's `__pycache__` are byte code
- Node.js (V8) compiles JS to internal byte code before running
- Provides a good balance: portable + faster than pure interpretation

## 3. Machine Code

### What is it?
Machine code is the **lowest-level code** — raw binary instructions that a CPU executes **directly**. There is no interpretation or translation step at runtime.

### Characteristics
- Binary format: sequences of `0`s and `1`s
- CPU-specific — x86 machine code won't run on ARM
- Extremely fast (no translation overhead)
- Generated from source code by a compiler (C, C++, Rust) or JIT from byte code
- Completely unreadable to humans

### How It Works

```
Source Code  -->  [Compiler]  -->  Machine Code  -->  CPU executes directly
hello.c            gcc            hello.exe / a.out
hello.cpp          g++            binary
hello.rs           rustc          binary
```

### Example — C Source to Machine Code

**C Source Code:**
```c
// hello.c
#include <stdio.h>
int main() {
    printf("Hello, World!\n");
    return 0;
}
```

**Compiled to Assembly (human-readable machine code equivalent):**
```asm
main:
    push    rbp
    mov     rbp, rsp
    lea     rdi, [rip + .LC0]    ; Load address of "Hello, World!\n"
    call    printf               ; Call printf
    mov     eax, 0
    pop     rbp
    ret
```

**Actual Machine Code (hex representation of binary):**
```
55 48 89 e5 48 8d 3d 00 00 00 00 e8 00 00 00 00 b8 00 00 00 00 5d c3
```
> That hex dump IS the machine code — binary `0`s and `1`s that the CPU reads.

### JIT Compilation (Just-In-Time)
Modern runtimes like V8 (Node.js) and JVM use **JIT compilation** — they convert byte code into machine code **at runtime**, giving near-native performance.

```
Source Code  -->  Byte Code  -->  [JIT Compiler at runtime]  -->  Machine Code
hello.js          V8 IR              V8 JIT                         CPU executes
```

## The Full Pipeline: From Source to Execution

```
You write:          greet("World")        <-- SOURCE CODE (human)
         |
         v
Compiler/Transpiler produces:
         |
         v
[Byte Code or AST]                        <-- BYTE CODE (intermediate)
    LOAD "World"
    CALL greet
         |
         v
VM / JIT compiles to:
         |
         v
01010101 01001000 10001001 11100101...    <-- MACHINE CODE (CPU)
         |
         v
CPU EXECUTES  -->  "Hello, World!" printed to screen
```

## Language Classification

| Language | Approach | Output |
|---|---|---|
| C / C++ | AOT Compiled | Machine code directly |
| Rust | AOT Compiled | Machine code directly |
| Java | Compiled + VM | Byte code → JVM → (JIT) Machine code |
| Python | Interpreted + VM | Byte code (`.pyc`) → Python VM |
| JavaScript | JIT Compiled | Source → V8 IR → JIT → Machine code |
| C# | Compiled + VM | Byte code (IL) → CLR → (JIT) Machine code |
| Go | AOT Compiled | Machine code directly |

## Key Differences — Plain English

### Source Code
> "The recipe written by the chef in English."

### Byte Code
> "The recipe translated into a standardized kitchen shorthand — any trained chef in any kitchen can follow it, but a random person still can't read it."

### Machine Code
> "The electrical signals sent directly to the oven, mixer, and stove. No translation needed — but only works with *that exact* kitchen hardware."

## Why This Matters for JavaScript Developers

When you write a Playwright test in Node.js:

1. **You write** `.js` source code → `test.spec.js`
2. **Node.js (V8 engine) reads** your source code
3. **V8 parses** it into an AST (Abstract Syntax Tree)
4. **V8 compiles** the AST into **byte code** (Ignition interpreter)
5. **V8's JIT compiler (Turbofan)** converts hot byte code into **machine code**
6. **Your CPU** executes the machine code → browser automation happens

This is why Node.js is fast despite JavaScript being a "scripting language" — it ultimately runs as machine code.
